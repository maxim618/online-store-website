
# ADR-005: Implement Idempotency in Payment Processing

**Status:** Accepted
**Date:** 2026-02-17

---

## Context

Подсистема платежей функционирует в распределённой среде, где возможны:

* повторные HTTP-запросы из-за таймаутов
* автоматические ретраи со стороны клиента
* сетевые сбои с неопределённым результатом операции

Без специального механизма повторный запрос на создание платежа может привести к:

* дублированию операций
* неконсистентному состоянию заказа
* финансовым рискам

Идемпотентность в контексте платежей является бизнес-требованием, а не инфраструктурной оптимизацией.

---

## Decision

Принято решение реализовать идемпотентность на уровне прикладной логики платежей.

### Основные принципы

* Каждый запрос создания платежа сопровождается `idempotency key`.
* Повторный запрос с тем же ключом не создаёт новый агрегат.
* Клиенту возвращается результат уже существующей операции.
* Идемпотентность интегрирована в жизненный цикл агрегата `Payment`.

Для хранения и координации запросов используется внешний operational storage — **Redis**.

Redis используется:

* как механизм координации повторных запросов
* как временное хранилище состояния обработки
* с ограниченным TTL для idempotency key
* но **не как кэш**

### Архитектурная граница

* Доменная модель знает о наличии `idempotency key`.
* Домен не зависит от Redis.
* Взаимодействие с Redis осуществляется на уровне service-слоя.

Service-слой:

* проверяет наличие операции по ключу
* обеспечивает атомарность проверки и создания операции
* инициирует создание агрегата только при отсутствии существующего
* сохраняет согласованное состояние

Идемпотентность рассматривается как часть бизнес-контракта между клиентом и системой.

---

## Alternatives Considered

1. Полагаться только на уникальные ограничения БД
   --> отклонено как недостаточно выразительное решение без явной бизнес-семантики.

2. Реализовать идемпотентность исключительно на уровне HTTP (например, через прокси)
   --> отклонено, защита должна существовать на уровне бизнес-логики, а не только транспорта.

3. Игнорировать проблему на раннем этапе
   --> отклонено из-за высокого риска для финансовых операций.

4. Использование Redis Lua scripts для обеспечения атомарности
   --> отклонено на текущем этапе как преждевременное усложнение.
   Текущая нагрузка не требует этого; архитектура допускает усложнение при росте требований.

---

## Consequences

### Positive

* Защита от повторной обработки платежей
* Устойчивость к повторным HTTP-запросам
* Снижение риска двойного списания
* Повышение устойчивости к сетевым сбоям
* Явная бизнес-семантика операции
* Подготовка к масштабированию

### Negative

* Зависимость от Redis
* Усложнение прикладной логики
* Необходимость управления временем жизни ключей
* Возможные гонки без атомарных Lua-операций
* Повышенные требования к консистентности хранилища

---

## Architectural Impact

Введение идемпотентности:

* усилило роль агрегата `Payment` как центра бизнес-инвариантов
* закрепило границу между доменом и инфраструктурой
* интегрировало Redis в архитектуру как координационный компонент
* повысило устойчивость системы к повторным запросам

Идемпотентность стала частью архитектурного контракта системы, а не вспомогательным техническим механизмом.
