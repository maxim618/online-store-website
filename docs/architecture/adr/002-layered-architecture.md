
# ADR-002: Migrate from DAO to Layered Architecture

**Status:** Accepted
**Date:** 2026-02-17

---

## Context

На раннем этапе проект использовал DAO-подход, характерный для классических Spring / Java EE приложений:

* прямое использование DAO
* смешение SQL, маппинга и бизнес-логики
* слабое разделение ответственности
* высокая связанность компонентов

Это приводило к:

* сложности тестирования
* усложнению эволюции доменной модели
* размазыванию бизнес-логики по слоям
* трудностям при внедрении новых механизмов (идемпотентность, state machine, Redis)

Текущая структура не обеспечивала чётких архитектурных границ.

---

## Decision

Принято решение отказаться от DAO-подхода и перейти к слоистой архитектуре:

```
Controller → Service → Repository → Entity
                         ↓
                        DTO ← Mapper
```

### Принципы слоёв

**Controller**

* принимает HTTP-запросы
* валидирует входные данные
* не содержит бизнес-логики
* не работает напрямую с Entity

**Service**

* реализует прикладные сценарии
* управляет транзакциями
* координирует доменные операции
> Под прикладным сценарием понимается завершённая операция системы, инициируемая
> внешним запросом и координирующая работу доменной модели и инфраструктуры.

**Repository**

* абстрагирует доступ к данным
* реализован через Spring Data JPA

**DTO**

* изолируют API от доменной модели
* предотвращают leakage Entity

**Mapper**

* отвечает за преобразование DTO ↔ Entity

Ручные DAO-классы удалены и заменены на декларативные репозитории Spring Data JPA.

---

## Alternatives Considered

1. Сохранить DAO-подход и постепенно рефакторить
   → отклонено из-за структурной несостоятельности подхода.

2. Перейти сразу к полноценной hexagonal architecture
   → отклонено на данном этапе как преждевременное усложнение.

---

## Consequences

### Positive

* Чёткое разделение ответственности
* Улучшенная тестируемость
* Снижение связанности компонентов
* Упрощение дальнейшего внедрения domain layer

### Negative

* Увеличение количества классов
* Необходимость поддерживать маппинг между слоями
* Дополнительная сложность при реализации простых сценариев

---

## Architectural Impact

Этот шаг стал структурной основой для:

* последующего выделения domain layer
* внедрения state machine в Payment
* реализации идемпотентности
* интеграционных тестов

Layered architecture сделала стабильный архитектурный каркас,
с помощью которого стало возможно дальнейшее развитие системы.
